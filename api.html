gr8.BI API Documentation

NOTE: This service is in PRIVATE BETA until further notice

This is the primary API documentation for the cloud services provided by gr8.BI. The service allows you to build data flows and jobs, using visual tools, and execute them in the cloud paying only for compute resources used. There are other resources avialable for getting started, and learning how to build your data flows to run on the gr8.bi service.

Some important overall concepts you should know about the API:

  • RESTful endpoints

    The API is a collection of RESTful endpoints, that allow for creating, running, and management of data flows in the cloud service.

  • SSL only

    The API is available through SSL only. HTTP request will give a hard error.

  • JSON only

    The API currently only supports interaction using JSON. We may support xml in the future, but it’s currently not on the roadmap. Additionally, there are some endpoints that accept Multipart data (file uploading, etc) but unless noted they are all standard JSON REST endpoints.

  • Unsupported but planned features

    JSONP support, gzip compression, API rate throttling, fine grained token access, parameters at [account|job] level, all list REST endpoints (get jobs), webhooks, HTML waiting pages, redirect URL support, object names, job grouping, timeout run support, encryption services, general purpose annotation kv tagging, metric add to Job Run (elapsed time, network stats, etc). Bundle versions, inline runJob (post ktr, createJob and run all in one request). Pretty JSON PDI Plugins/marketplace. Different Kettle versions. Talend, CoverETL support.

  • Unsupported and not planned features

    XML support, SOAP

Security 

Authentication

There are two methods for authenticating to the API. Email and Password are used for senstive operations (account and tokens) and tokens are used for all other operations (running jobs, etc).

  • Username and Password (Account and Token)

    Using basic HTTP auth header (Authorization: Basic), over SSL, one can use the account owner (email) and password (mypassword) to authenticate and authorize against all Account endpoints. Using this method is akin to having “root” access and is only allowed when necessary (managing tokens, and accounts). Everything in the “Account” group requires HTTP Basic Auth, everything else is tokens.

  • Tokens (Jobs, JobBundles, and Runs)

    Tokens are the primary method for using all endpoints. Tokens can be sent via a header (X-Auth-Token), URL parameter (token). They are scoped to grant authorization at the same full scope as the original account owner, so anyone who has a token has similar and equal access to resources on the account. Tokens can be managed as documented below. Tokens will eventually have operation (Read Only, Modify, etc) restrictions along with scope (only for this particular job).

Tokens are JWT tokens; eventually we intend to allow per-account JWT signing and fine grained resource access.

Unauthorized or Unauthenticated

If you try to access the service with invalid credentials, expired tokens or try to perform operations on objects you don’t have access to you will receive a JSON unauthorized object. We’ll return code 401, and an object that will provide some description of what went wrong:

{
  "code": "401",
  "message": "Unauthorized",
  "description": "Invalid token"
}

API General 

Object Identifiers

All objects (Accounts, Runs, Job Runs, Schedules) have a unique ID, that both identifies the object and type. The ID is a 12 digit string (except tokens which are longer). The first 3 digits is the “type” of object identifier, followed by a dash, then a code ([0-9a-zA-Z]). The ID for the object is the entire string, including the object type.

The following is the dictionary of types, and prefixes:

  • Account (act) act-12345678
  • Job (job) job-12345678
  • Job Bundle (bun) bun-12345678
  • Job Run (run) run-12345678

Redirect

All endpoints support redirect URL paramters that will be used, instead of returning the original JSON response. The response object that would have originally been returned is “query-string-i-fied” using the qs package. Parameters are a little “funny” on the redirect, but that’s the nature of taking a complex object (JSON) and making it into key/value pairs.

  • redirect_success - Is used when the operation was successful, and a normal 200 response was about to be returned. It redirects to whatever URL was provided using response code 302, appending the return object on as paramters using qs as mentioned above.
  • redirect_error - Is used when the operation failed. The failure could be because of an error, or an authentication issue. The redirect be to the URL location provided using response code 302, and will append the return object on as paramters using qs as mentioned above.

Query Syntax

Listing endpoints support filtering, limiting, and ordering of results through URL parameters.

  • limit - Maximum number of records to return. Default: 10.
  • q - Is a mostly Lucene style query, that serves to limit the result set. There are a very limited set of fields available for querying, and all fields are indexed in their entiretiy (“My Job Name” won’t match a query “Job”). The short and skinny, a query would look like q=run_status:work.complete. Default: Nothing
  • order - Allows different sort ordering. Also follows Lucene ordering syntax. _Default: -created_at<string>

Calling a list endpoint with none of these parameters would return the ten most recently created Jobs/JobBundles/Runs/etc.

All objects have the following indexed and available for querying:

  • default - The id of the object (bun-12345678, run-12345678)
  • created_at - An ISO Date string (suitable for collation/ordering) 2014-05-26T22:00:47.676Z

Individual /list endpoints in the API below indicate which additional fields they have available for use in q and order.

Errors

Errors from either user submission issues (missing paramters, etc) or from actual service issues (EEEKS!) return an error JSON object, along with response code 401.

We need to document error codes (eventually) but hopefully the message is enough to provide a useful amount of debugging:

Token 

Tokens are the way to access the API endpoints and authorize and authenticate. Currently, we have global tokens (tok- prefixed) but plan to support JWT signing (per account scoped tokens) in the future.

Create Token 

Create TokenPOST /act/token

Creates a new token. Any token grants all access to all resources for the users account (act_id). The act_id for the token is determined from the Email Address used in the HTTP Authorization Headers. Expire time is calculated from the current time, adding the expire_seconds_from_now number of seconds. If omitted, (30*60) is the default (30 minutes).

      Job Bundle 

      JSON 

      Job Bundles are zipped up definitions of Data Flow code. For instance, for Kettle the bundle would contain Transformation files (.ktr) and Job Files (.kjb). Bundles may also contain other resources (such as template .xls files, etc) but don’t contain any transient data. The Bundles are the “code” of your Data Flow solution.
