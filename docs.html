Getting Started Fast 

We assume you have access to curl, and you’ve been emailed that your account is active in the PRIVATE BETA. Our CLI is bash/curl based you so need to be on OSX or Linux.

Step 1 (Download CLI/Examples)

You can get the CLI through git (git clone https://github.com/nagoodman/gr8bi_extras.git) or by downloading a zip of the latest: (https://github.com/nagoodman/gr8bi_extras/archive/master.zip). We’ll refer to this directory as GR8_BI_EXTRAS_LOCATION

Step 2 (Install CLI)

Head to the directory GR8_BI_EXTRAS_LOCATION/cli and run the following command to initialize the CLI

./gr8_CONFIGURE.sh "youremail@company.com" "password"

You should receive a JSON response like this:

{
  "token": "tok-abcd1234ABCD",
  "expires": "2014-05-18T05:26:41.823Z",
  "act_id": "act-12345678"
}

This will initialize the CLI and place a file (~/.gr8bi_token) that authenticates the CLI for use with the service 30 days.

Step 2 (Upload and Run Example)

Head to GR8_BI_EXTRAS_LOCATION/cli and run the following command:

./gr8_pushAndRun.sh "../examples/output" "output.kjb" ""

You should see a variety of output. The gr8_pushAndRun.sh command does the following steps on your behalf:

  • Zips the sample job (../examples/output) into a Job bundle.
  • Uploads the Job Bundle to gr8.BI
  • Creates a Job for execution (that includes the location in the file to run) output.kjb
  • Runs the Job with no parameters (outpout.kjb needs no runtime parameters).

NOTE the run_id (run-12345678); we’ll use that in Step 3.

Step 3 (Retrieve Run Status)

Execute the following commands to get updates on the run status for your run-id.

./gr8_getRun.sh "run-12345678"
{
  "run_id": "run-12345678",
  "run_status": "work.complete",
  "job_id": "job-12345678",
  "params": {},
  "created_at": "2014-05-17T17:34:15.871Z",
  "log_url": "https://url.log",
  "output_url": "https://url.zip",
  "exit_code": 0,
  "exec_seconds": 25
}

Refer to the Get Run API documentation for specifics on the values returned. Of particular note is:

  • run_status : This will indicate if the run is scheduled, running (work.started), or completed (work.complete).
  • log_url : This is a URL to view the log file. The log file is updated periodically and will be available approximately 60 seconds after initial execution. You can view the log in a web browser or view it via curl "https://url.log".
  • exit_code : If a run is a complete, this is the overall job completion status. 0 indicates successful run, other values indicate some sort of error (check log_url).

Getting Started API 

Prerequisites

This guide assumes you have a basic knowledge of ETL, Kettle and how to create a Kettle jobs and transformations. You can use one of our example jobs to test the service, but to do any real work, you’ll need to download, learn and use Kettle.

Additionally, we assume you have a data flow written in Kettle that you are ready to execute on our cloud service. If you’d like a start with our testJobBundle, you’re welcome to and skip to Step 2.

We assume you have access to curl.

We will use act-12345678 and tok-abcd1234ABCD during the rest of this getting started guide; however, we assume that you would substitute the actual account id and token you received during private beta enablement.

Step 1 (Prepare your Data Flow)

  • Zip up all files you need as part of your job so that all the KTR and KJB needed to run your job (along with any other templates, etc) are in a zip.
  • Make sure to note (use unzip -l if needed) the location in the zip file for the master KJB file you want started. Often there are several KJB files in a bundle, you need to have the location of the job (path_in_zip/myjob.kjb) you want executed.

Step 2 (Upload your Data Flow)

The zip file that contains your Data Flow is called a Bundle. You can upload it via curl using the following command (replace act-12345678 with your account, tok-abcd1234ABCD with your token, and bundle.zip with the name of your zip file you created in Step 1).

curl -X POST \
--data-binary @bundle.zip \
-H 'Content-Type: application/zip' \
-H "X-Auth-Token: tok-abcd1234ABCD" \
https://api.gr8.bi/current/act-12345678/jobBundle

You should receive a response that includes the bun_id for your newly created bundle. You’ll need this for Step 3! We’ll refer to this id as bun-12345678 in subsequent steps

The full documentation for upload bundle is in our API documentation.

Step 3 (Create a Job)

Let’s create the job, so it’s ready for execution! You can create the job by executing the following curl command:

curl -s -X POST \
-H "Content-Type: application/json" \
-d '{ "bun_id" : "bun_id_from_STEP2","job_entry_point" : "path_in_zip/myjob.kjb"}' \
--header "X-Auth-Token: tok-abcd1234ABCD" \
"https://api.gr8.bi/current/act-12345678/job"

If you are using the testJobBundle.zip, the job_entry_point is test-job.kjb.

You should receive a job_id for the newly created job. We’ll use that in step 4, and refer to it as job-12345678.

The full documentation for create job is in our API documentation.

Step 4 (Run your Job)

Let’s run our job! Excute the following curl command:

curl -s -X POST \
  -H "Content-Type: application/json" \
  -d '{ "job_id" : "job-12345678"}' \
  --header "X-Auth-Token: tok-abcd1234ABCD" \
  "https://api.gr8.bi/current/act-12345678/run"

Most Data Flows start execution within 60 seconds!

If your Data Flow causes any affects (FTPs a file, pushes to S3, etc) it shouldn’t be long before you see those affects! We’ll also check the status and logs in the next step!

You should receive a run_id, along with the current state of the run. We will use this run_id to check the status, and access the log file.

The full documentation for run job is in our API documentation.

Step 5 (Check status and Retrieve Logs)

Let’s check the status of the run! Execute the following curl command:

curl https://api.gr8.bi/current/act-12345678/run/run-12345678?token=tok-abcd1234ABCD

You should receive a JSON response that includes details about the run:

{
  "run_id" : "run-12345678"
  ,"run_status" : "work.complete"
  ,"created_at": "2014-04-14T21:46:08.568Z"
  ,"job_id" : "job-12345678"
  ,"log_url" : "https://biglongURL.com"
}

If your “run_status” is work.complete then you should also see a log_url that has the log file for your job run. You can paste this URL into a web browser or use curl again to review the log:

curl https://biglongURL.com

If your run status is scheduled.ready your run is waiting to run on one of our machiens. If it’s work.started your Data Flow is running, but has not completed yet.

The full documentation for get run is in our API documentation.

Runtime 

Kettle Environment

Variables

Variables have been set GLOBALLY for the entire data flow run prior to your data flow executing. Some are special variables set by the data flow service:

  • BI_GR8_OUTPUT_DIR - This contains the location of the output directory. Any files placed in this directory, will be zipped and saved at the completion of the run
  • BI_GR8_SMTP_HOST - The HOSTNAME for sending SMTP via gr8.BI provided account.
  • BI_GR8_SMTP_PORT - The PORT for sending SMTP via gr8.BI provided account.
  • BI_GR8_SMTP_USER - The USER for sending SMTP via gr8.BI provided account.
  • BI_GR8_SMTP_PASSWORD - The PASSWORD for sending SMTP via gr8.BI provided account.
  • BI_GR8_RUN_PAYLOAD - The location of the file on the machine, that has binary data if provided with the run. For instance, if you performed an HTTP PUT of an excel file, the file is now on the server on the filesystem at the location contained in the variable.

In addition to the standard data flow service variables, any data flow params passed to the web services to create runs will also be available. For instance, you if included “key1=value1” on the URL when running a job then key1 will be defined and available globablly

Version

gr8.BI currently runs a standard version of Kettle 5.0.1 CE Edition. We plan to support multiple versions of Kettle in the future; we also reserve the right to run on a slightly modified version of Kettle in the future.

Hardware and OS

Your Data Flows will typically need not interact with the environment directly. Kettle, as a Java project, has very little dependency on the OS for libraries. Even as such, here is some information about the OS:

  • Debian 7 Wheezy
  • 64 Bit
  • Oracle JDK 7u45 Installed

While the most important aspect will be how well your Data Flow performs, we believe we are providing a high performance Data processing node suitable to almost all Data Flow tasks. Data Flows currently run on the following virtual server machines:

  • Modern cloud server, with 2 virtual core.
  • 1.8 GB of RAM (1.5 GB Java Heap)
  • 10 GB Disk Space (8 GB usable by flow)

NOTE: We currently run on a google compute engine in the US us-central1-a but have future plans for other providers and APAC/EMEA.

Limitations and Quotas 

Data Flow Limitations

The following are limitations when running Data Flows

File and Directory limitations

You are limited to reading and writing on the local filesystem to:

  • /tmp/ (java.io.tmpdir) for temporary sorting/etc
  • /grbi/customer/* for any and all file system access, temp files, and output files.

Network Limitations

  • No outbound SMTP traffic on port 25. gr8.BI provides a set of credentials (quota-ed and monitored) for sending outbound emails. Kettle/Variables section for specifics.
  • No inbound connections of any kind are allowed.

KJB/KTR Limitations

  • No access to system executables are allowed. Any Data Flow step that needs access to the OS for execution (bulkloaders, etc) will fail with a hard error.
  • Job Bundles can contain any number of files, and subdirectories. The total combined size of the bundle can not exceed 20MB (zipped).
  • Database logging should NOT be enabled! Kettle allows you to setup logging to a database, our service will eventually collect and manage this for Data Flows so your KTR and KJBs should not use/require this functionality.

kettle.properties or shared.xml

  • Currently we do not support user created kettle.properties and shared.xml files. You can create your own properties file (or .yaml) file, include it with your bundle, and set variables at the start of your data flow as a workaround.

Service Quotas

Run

The following are the default quotas for using the gr8.BI service.

NOTE: These are standard limits for PRIVATE BETA; contact us if you need more. If the service emerges from PRIVATE BETA to a paid service, paid plans will offer much larger capacities.

  • API Calls Monthly : Each account is limited to 150,000 API calls per month.
  • API Calls per minute : Each account is limited to 250 API calls per minute. Additionally, each IP is limited to 250 API calls per minute.
  • Data Flow Minutes Monthly : Each account is limited to a total of 25,000 ETL minutes per month.
  • Data Flow Concurrency : Each account is limited to having 5 jobs running at any given time; additional jobs are queued and will be executed once other jobs have completed.
  • Bandwidth Incoming Monthly : Each instance is allowed to consume as much incoming bandwidth (from Internet to the Data Flow service) as desired.
  • Bandwidth Outgoing Monthly : Each account is limited to 200GB of outbound traffic. Outbound traffic is all traffic from Data Flow service outbound, and access of URLs provided by the service (such as log_url, output_url, etc).
  • Other Things We Haven’t Thought of : We’ll add to this list as we understand how people are using the service and we need to ensure fairness and service availability.
